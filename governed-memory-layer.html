// motivation

The problem with naive shared memory for agents

Multi-agent systems need some kind of shared state. The obvious implementation is also the dangerous one: let each agent append what it thinks is important into a common memory store and hope the useful state rises above the noise. In practice, this fails in a very specific way. Bad memory is durable. A vague note becomes a tracked issue. A paraphrased lifecycle update closes the wrong row. A malformed result record survives long enough to mislead the next agent.

The hard part is not storing text. The hard part is deciding what a note is allowed to mean once it becomes shared canonical state. That means filtering noise, preserving identity across time, handling contradictions, and refusing to guess when a later note refers ambiguously to an earlier issue, constraint, or decision.

The central question: how do you let agents accumulate durable shared memory without allowing ambiguous or malformed writes to silently corrupt the canonical state everyone else depends on?
// architecture

How the governed memory layer works

The system imposes a strict separation between private note capture and shared-memory mutation. Agents are allowed to write raw notes into working memory. They are not allowed to write canonical shared state directly. Every note passes through a governance stack before anything becomes durable shared memory.

+------------------------------------------------------------+ | AGENT / TOOL OUTPUT | | Natural-language notes, observations, status updates | +------------------------------------------------------------+ | v +------------------------------------------------------------+ | WORKING MEMORY | | Private, per-agent, per-run note buffer | | | | Stores timestamped notes only. | | No schema enforcement. No shared writes. | +------------------------------------------------------------+ | v +------------------------------------------------------------+ | PROMOTION PIPELINE | | | | 1. Interpreter: should this note become shared memory? | | 2. Resolver: what existing thing, if any, does it refer to?| | 3. Validator: is this write structurally safe? | | 4. Inputter: commit to ledger and project canonical state | +------------------------------------------------------------+ | +-----------------------------+ | | v v +------------------------------------------------+ +----------------------+ | EVENTS LEDGER | | PENDING QUEUE | | append-only events_memory | | pending_memory_events| | every committed canonical write is recorded | | unresolved but | | first, even if projection fails | | important lifecycle | | | | writes wait here | +------------------------------------------------+ +----------------------+ | v +------------------------------------------------------------+ | DETERMINISTIC PROJECTOR | | shared_memory_writer.py | | | | Updates shared_plan / shared_issues / shared_constraints | | / shared_decisions / shared_results / shared_task_state | | / shared_learnings | +------------------------------------------------------------+ | v canonical shared memory

The four governance stages

  • 01
    Interpretation - classify a note as meaningful shared state or reject it as noise, speculation, or insufficiently structured
  • 02
    Resolution - decide whether a lifecycle note safely binds to an existing canonical entity, creates a new one, or must remain provisional
  • 03
    Validation - enforce bucket, operation, payload, and lifecycle-target correctness deterministically before canonical mutation
  • 04
    Projection - persist the ledger event, then update canonical tables through deterministic SQL rather than free-form model output

The key architectural claim is not that the interpreter is always right. It is that shared state is never mutated on the basis of model text alone. The model can propose meaning. It cannot directly define canonical memory.

// implementation

What a governed write actually looks like

The most important interface in the system is the structured write request emitted after interpretation. From that point on, the pipeline becomes progressively more deterministic. 

// Example: a note becomes a structured write request
{
  "decision": "accept",
  "bucket": "issues",
  "target_id": "pandas_import_blocker",
  "operation": "resolve",
  "payload": {},
  "reference_text": "the earlier blocker",
  "candidate_aliases": ["pandas error", "import blocker"],
  "confidence": 0.84,
  "rationale": "lifecycle update referring to an earlier open issue"
}

The resolver then compares that request against current canonical context: open issues, active constraints, and active decisions. If the reference clearly binds to one existing row, the write can commit. If it cannot safely identify the target, the system does not guess. It preserves the write in pending_memory_events and retries later when fresh canonical state exists. 

// Example: what gets durably recorded on commit
{
  "event_id": "uuid",
  "bucket": "issues",
  "target_id": "pandas_import_blocker",
  "operation": "resolve",
  "payload_json": "{}",
  "raw_input": "The earlier blocker is fixed.",
  "source_ref": "{ matched_target_id, candidate_matches, resolution_reason }",
  "applied_successfully": 1
}

That ledger row is the durable source of truth. Canonical tables are downstream projections. This is why the system can recover from projection failures: the audit trail exists independently of whether the projector succeeded in that moment.

// evaluation

Benchmark coverage

The repository includes a dedicated benchmark harness rather than only unit tests. The benchmark does not score generated text. It scores the final shared-memory snapshot: whether the right canonical rows exist, whether stale rows remain, whether pending debt accumulates, and whether replay can reconstruct state after projection failures.

The harness compares the governed system against simpler baselines such as append-all memory and no-shared-context classification. The current benchmark registry contains 304 trajectories across six families.

Family Description Count What It Stresses
A Promotion filtering 64 Noise rejection and selective promotion
B Lifecycle transitions 48 Open -> resolve, active -> invalidate, latest task state
C Duplicate avoidance 48 Identity continuity across aliases and paraphrases
D Conflict handling 48 Authoritative override and stale-state cleanup
E Cross-bucket coherence 48 Agreement between issues, constraints, decisions, results, and task state
F Replay robustness 48 Ledger durability and reconstruction after projection failure

The evaluation focus is architectural, not cosmetic. Family A asks whether the system keeps junk out. Families B through E ask whether shared memory stays coherent over time. Family F asks whether the ledger is sufficient to recover the state after deterministic projection fails.

// key finding

The hardest problem is not storage. It is identity continuity.

Schema validation is comparatively easy. The genuinely difficult problem is preserving the identity of a thing across time when agents stop referring to it the same way. A system may correctly capture an issue today as pandas_import_blocker, then encounter tomorrow's note phrased only as "that earlier blocker."

FINDING

The architecture's main safety move is to turn uncertainty into explicit state instead of silent corruption

When a lifecycle note cannot be bound confidently to an existing canonical row, the system does not force a guess. It records the unresolved write in a pending queue and retries later after canonical state changes. This is a stronger architectural move than it first appears. It converts ambiguity from a hidden failure mode into a visible, inspectable backlog.

In other words, the design treats deferred resolution debt as healthier than incorrect certainty. That choice shapes the entire system: resolver logic, reference-memory accumulation, replay behavior, and benchmark metrics all exist because of it.

This is why the append-only ledger and deterministic projections matter together. The ledger preserves what happened. The canonical tables preserve what is currently believed. The pending queue preserves what the system suspects but refuses to collapse prematurely.

// failure modes

How simpler memory systems fail

The repository includes baseline harnesses precisely because the failure modes are predictable. Once you compare governed memory against simpler designs, two recurring breakdowns explain most of the difference.

F1

Over-promotion

In append-all memory, every note becomes durable state. Internal musings, transient telemetry, and speculative side comments all survive as if they were canonical facts. The result is memory pollution: extra issues, fake learnings, and contradictory state rows that future agents must reason around.

F2

Lifecycle misbinding

In note-by-note systems without shared-state context or reference memory, later updates frequently attach to the wrong thing or create duplicates instead of closing the original row. The visible symptom is not only duplicates; it is stale active state that never gets retired.

The governed system does not eliminate all uncertainty. What it does is force that uncertainty through explicit boundaries: interpretation, resolution, validation, and deterministic write semantics.

// scope

Limitations and current scope

This is a memory architecture and benchmark suite, not a full production multi-agent platform. The repository is intentionally concentrated on the note-to-memory boundary and the correctness properties around that boundary.

The following are current scope limits, not accidental omissions:

  • -
    Identity-specialized resolver logic exists only for issues, constraints, and decisions
  • -
    No full runtime agent orchestration layer or service API is included yet
  • -
    Replay is implemented through the fault-injection benchmark harness, not a standalone production repair tool
  • -
    Classification quality still depends partly on the live interpreter model and prompt
  • -
    The system is synchronous and SQLite-based rather than distributed
